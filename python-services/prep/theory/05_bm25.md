After prev steps, ChromaDB holds dense vectors (embeddings) for all 40 QA
pairs. Dense search is great for semantic queries — "what causes
memory corruption" finds docs about heap corruption even if those
exact words aren't used.

But dense search has a blind spot: exact keyword matching. If someone
asks about ft_printf %d flag, the embedding for %d has almost no
signal because subword tokenizers barely know what it means. A model
trained on text has weak representations for punctuation-heavy tokens,
format specifiers, function names.
===================================
 BM25 (Best Match 25) fills is a keyword-scoring algorithm
 ==================================
  — when your query contains ft_printf, BM25 finds every doc that has
  ft_printf and scores them by:

  BM25(q, d) = Σ IDF(qi) × tf(qi,d) × (k1+1)
                            ─────────────────────────────
                            tf(qi,d) + k1 × (1 - b + b × |d|/avgdl)

  Where:
  - IDF: how rare the term is across all docs — rare terms matter more
  - tf: how often the term appears in this doc
  - k1 = 1.5: term saturation — after a word appears a few times, extra
  occurrences add diminishing returns (unlike raw TF where 10×
  appearances = 10× score)
  - b = 0.75: length normalization — longer docs don't automatically win
   just because they contain more words
  - avgdl: average document length across the corpus

  This is the same formula Elasticsearch uses. rank_bm25 implements it
  in pure Python with the BM25Okapi variant (the standard). No I/O, no
  network calls — pure math on strings.

  Why keep both? Dense catches semantics, BM25 catches exact tokens.
  Step 5 (retriever.py) will merge both result lists using RRF.
  ==============================
 Design decisions in bm25_index.py

  Tokenization: We use .lower().split() — simple whitespace
  tokenization. For Q&A pairs that are already clean, structured text,
  this is sufficient. The complex subword tokenization from py_rag.py
  (tokenize() with regex punctuation stripping) would hurt here:
  ft_printf should stay as one token, not get split.

  Score filtering (if s > 0): BM25 gives a score of 0.0 when a query
  term appears in zero docs in the corpus. Returning zero-score results
  to the RRF merger is noise — they don't add information, they just
  dilute. We filter them out.

  Guard on unbuilt index: If search() is called before build(),
  returning [] is the correct safe behavior — the merger in retriever.py
   will just get an empty sparse result and fall back to dense-only. No
  crash.

  Error on empty corpus: build([]) raises ValueError immediately rather
  than silently creating a useless index.
