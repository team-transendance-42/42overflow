  Caddy handles TLS termination — meaning:

  1. The browser connects to Caddy over HTTPS (port 8443), with an encrypted TLS connection
  2. Caddy decrypts (terminates) that TLS connection
  3. Caddy then forwards the request to SvelteKit over plain HTTP internally (port 5173)

  This means only Caddy needs to manage certificates. The internal services (SvelteKit, Go llm-server, Python services) communicate over unencrypted
  HTTP on the local network/container network, which is fine since they're not exposed to the public internet.

  "Termination" = the TLS encryption ends at Caddy; it doesn't pass through to the backend.

  Go llm-server middleware stack (applied to all routes):
    ErrorRecovery → InternalSecret (X-Internal-Secret header) → RateLimiter


  Python RAG admin routes (require X-Admin-Token):
    POST /admin/sync-chroma   → read Postgres → sync to ChromaDB → rebuild indexes
    POST /admin/seed-postgres → insert fake test data into Postgres
    POST /admin/wipe          → delete all Postgres data + ChromaDB docs
    GET  /admin/metrics       → retrieve_total, errors, corpus_size, uptime

  Caddy terminates TLS and reverse-proxies everything to SvelteKit.

  SvelteKit is both the frontend and a BFF (Backend for Frontend). Page routes (/login, /subjects, /posts, etc.) render server-side with Prisma. API
  routes (/api/*) are thin proxies or direct DB handlers. Auth is handled by Better Auth — sessions stored in Postgres.

  Go llm-server is the LLM gateway, guarded by an X-Internal-Secret header and a rate limiter. It exposes three AI endpoints:
  - /api/ai-assist → streams from Gemini (external API)
  - /api/ollama → streams from local Ollama (gemma3:4b)
  - /api/community → RAG flow: check in-memory cache → call Python RAG → gate on confidence → stream from Ollama → cache the answer

  Python RAG service builds two in-memory indexes on startup (BM25 for keywords + NumpyIndex for cosine similarity). On each /rag/retrieve call it
  runs a hybrid search fusing both scores with RRF, returns top-4 context docs. ChromaDB is only used during admin sync — live retrieval is fully
  in-memory.

  Python STT service receives a multipart audio file, saves it to a temp file, runs faster-whisper (small model, CPU), returns the transcript as
  JSON.

  Ollama runs gemma3:4b locally — used by both the plain Ollama endpoint and the RAG community endpoint.
